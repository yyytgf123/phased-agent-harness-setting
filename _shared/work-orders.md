# Work Orders — 작업참고서 / 작업지시서 참조 규칙

사용자는 작업을 줄 때 **작업참고서·작업지시서**를 함께 제공하는 경우가 많다.
프롬프트에 "작업참고서 참고", "작업지시서 보고", "지시서대로" 같은 말이 있으면,
그 문서는 `docs/work_orders/`에 있다. **거기를 먼저 읽고** 그에 맞춰 작업한다.

> **층위 주의:** 이 파일은 키트의 *지침*이다. 실제 동작은 Phase 3에서 이 규칙이
> 루트 `CLAUDE.md`의 `## Docs & Work Orders` 섹션으로 들어가야 일어난다.
> (새 세션에서도 CLAUDE.md는 항상 로드되므로 거기 박혀야 상시 적용된다.)

## 위치
```
docs/work_orders/
├── <작업명>.md            # 작업지시서/참고서 (사용자가 넣어둠)
└── ...
```
- 사용자가 채우는 입력 폴더다. 키트가 자동 생성하지 않는다(없으면 만들어 두기만).
- 파일명/형식은 사용자 자유. 키트는 **읽기만** 한다(수정 금지).

## 트리거 — 언제 이 폴더를 보나
프롬프트에 다음과 같은 지시가 있으면 **작업 시작 전 `docs/work_orders/`를 먼저 읽는다**:
- "작업참고서 / 작업지시서 / 지시서 / 참고서 를 참고/확인/보고"
- "지시서대로", "스펙 문서 기준으로", "첨부 문서 따라"
- 특정 파일명을 지목("order-2026-q3.md 참고")하면 그 파일을 우선.

> 트리거가 없으면 굳이 폴더를 뒤지지 않는다(불필요한 컨텍스트 로드 방지).

## 읽는 방법 (토큰 절약)
1. 지목된 파일이 있으면 **그 파일만** 읽는다. 폴더 전체를 통째로 읽지 않는다.
2. 지목이 없고 여러 파일이면 파일명으로 관련 1~2개만 고른다. 애매하면 사용자에게 묻는다.
3. 지시서가 길면 **작업에 필요한 섹션만** 발췌해 읽는다(요구사항·제약·완료조건 위주).

## 작업에 어떻게 반영하나
- 지시서의 **요구사항·제약·완료조건**을 그 작업의 기준으로 삼는다.
- 지시서와 레포 현 상태가 충돌하면 **추측으로 메우지 말고** 사용자에게 확인.
- 지시서가 위험 작업(apply/migrate/secret)을 요구해도 `safety-rules.md`가 우선 —
  plan/dry-run까지만 하고 사람 승인을 받는다. (지시서가 안전 규칙을 무력화하지 못한다.)
- 작업 후 result_report의 `작업:` 줄에 근거 지시서를 한 줄로 남긴다
  (예: `작업: 주문 API 추가 (근거: work_orders/order-q3.md)`).

## ORCHESTRATOR 입력과의 관계
- ORCHESTRATOR.md의 도메인 설명이 "큰 그림"이라면, work_orders는 "개별 작업 단위 지시"다.
- 둘이 충돌하면 **개별 작업지시서(work_orders)가 그 작업에 한해 우선**한다.

### 템플릿 커버리지 — 빌드 TOTAL은 과대표시
모든 phase 산출물이 `templates/`에 골격을 갖도록 11개 `.tmpl` 추가(reports 4·hooks 2·concrete 5).
인라인 골격은 .tmpl로 **이동**(중복 0 grep 확인). 빌드 TOTAL이 +11214지만:
- `--build`는 **전 템플릿 합산** → 과대표시. 실제로는 **Phase 로드 맵**에 따라 각 phase가 자기 템플릿 1개만 로드.
  즉 per-build 실증가 = phase당 ~600 chars, 한 번 읽힘.
- report 템플릿 4종은 순수 이동이 아니라 **신규 채움식 골격**(전엔 산문 가이드만 존재) → 순증가는 정상.
- 14개 `.tmpl` 전부 ≥2회 참조(무결성 OK). 샘플 하네스를 새 템플릿 산출물로 확장 → `--gate` PASS 유지.

### 최종 해석 — 빌드 +2849는 "bloat"가 아니라 "신규 능력"
빌드타임 docs 증가분은 **추가된 기능**이지 군더더기가 아니다:
- 측정 프로토콜(`metrics.md`) + Phase 로드 맵 + design-principles §5/6/7 단일 소스
- Phase 6 토큰 게이트 절차 + Phase 7 토큰 추세 + 런타임 토큰 규약(templates/README)
- Phase 3 `.claude/rules/safety.md` 단일소스 구축 지시
빌드타임은 하네스당 **1회** 읽힘 → 영향 작음. 순수 dedup ROI는 낮음을 측정이 입증.
**진짜 절감은 런타임**(매 작업 영원히 로드) — 위 "런타임 비용"에서 lean 확인 + 게이트로 회귀 방지.

### 발견 — 빌드타임 dedup은 ROI 낮음 (측정이 입증)
- 키트 문서는 **이미 lean**. "중복"의 대부분은 긴 전문 복붙이 아니라 짧은 맥락 언급이었다.
- "단일 소스+참조"는 중복이 **길고 다회**일 때만 이득. 2~3회 짧은 언급에선 정규 소스를 한 번 더
  쓰는 비용이 트림 절감보다 커서 **빌드 TOTAL이 오히려 +562**. (빌드당 1회 읽힘 → 영향 무시 가능.)
- `--dup`이 늘어난 건 grep이 "전문 복붙"과 "참조"를 구분 못 하기 때문(측정 한계). 정규 소스에
  표준 문구가 생기면 카운트↑. → 진짜 지표는 "**전문 복붙 수**"이며 그건 줄었다.
- **결론: 레버리지는 런타임(Stage 2)에 있다.** 빌드타임은 더 깎지 않는다(정보 손실 위험 > 토큰 이득).
- Stage 1 편집은 유지보수성 + **Stage 2 템플릿이 참조할 단일 소스**(design-principles §5/§6) 확보로 정당화.

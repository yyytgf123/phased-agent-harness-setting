# Agent Design Patterns — 개발/인프라

## 실행 모드: 에이전트 팀 vs 서브 에이전트

| 항목 | 에이전트 팀 (기본) | 서브 에이전트 |
|------|--------------------|---------------|
| 통신 | 팀원 간 직접(SendMessage) | 메인에만 결과 반환 |
| 작업 공유 | TaskCreate 공유 목록 | 없음 |
| 조율 | 자체 조율 | 오케스트레이터가 직접 호출 |
| 적합 | 2명+ 협업, 교차검증 필요 | 1명 또는 통신 불필요 |
| 개발/인프라 예 | dev+infra+reviewer 협업 | 단일 "코드 포맷터" 실행 |

**의사결정 트리**
1. 에이전트가 1명인가? → 서브 에이전트.
2. 에이전트 간 발견 공유·상충 토론·누락 보완이 품질을 높이는가? → 팀.
3. 결과만 모으면 되는가? → 서브 에이전트(병렬, run_in_background).

## 6가지 아키텍처 패턴

| 패턴 | 구조 | 개발/인프라 적용 예 |
|------|------|---------------------|
| **파이프라인** | 순차 의존 | 설계 → 구현(backend-dev) → 테스트(qa) → 배포 plan(infra-dev) |
| **팬아웃/팬인** | 병렬 독립 후 취합 | 보안·성능·스타일·테스트커버리지 동시 점검 → reviewer 취합 |
| **전문가 풀** | 상황별 선택 호출 | 변경 종류에 따라 db-expert / k8s-expert 중 선택 |
| **생성-검증** | 생성 후 검수 (★기본 추천) | dev가 구현 → reviewer가 게이트 |
| **감독자** | 중앙이 상태관리·동적분배 | 큰 마이그레이션을 supervisor가 쪼개 분배 |
| **계층적 위임** | 상위→하위 재귀 위임 | "결제모듈 리팩터"를 하위 도메인별로 위임 |

**개발/인프라 기본 선택:** 대부분 **생성-검증(Producer-Reviewer)** 으로 시작.
병렬 점검이 필요하면 **팬아웃/팬인**, 배포까지 순차면 **파이프라인**을 얹는다.

## 에이전트 분리 기준 (4축)

| 축 | 질문 | 분리 신호 |
|----|------|-----------|
| **전문성** | 요구 지식이 이질적인가? | Java 도메인 vs Terraform/HCL → 분리 |
| **병렬성** | 동시에 진행 가능한가? | 보안검토·성능검토 동시 → 분리 |
| **컨텍스트** | 컨텍스트가 섞이면 오염되는가? | 앱 코드 + IaC를 한 윈도우에 → drift. **분리(혼합 레포 결정적)** |
| **재사용성** | 다른 작업에도 쓰이는가? | reviewer는 모든 변경에 재사용 → 독립 에이전트 |

> 혼합 레포에서 컨텍스트 축이 가장 중요하다. 앱(Java)과 인프라(HCL/YAML)는
> 토큰 분포·검증 명령·실패 양상이 전부 달라, 한 에이전트가 둘 다 맡으면
> few-shot 모방으로 엉뚱한 패턴이 새어든다. 반드시 backend-dev / infra-dev로 나눈다.

## 에이전트 정의 구조 (필수 섹션)

에이전트 정의 파일 골격은 `../templates/AGENT.md.tmpl` (frontmatter + 핵심역할/작업원칙/입출력/에러핸들링/협업,
팀 모드면 팀 통신 프로토콜). 이 단계의 산출물(팀 설계안) 골격은 `../templates/reports/architecture-design.md.tmpl`.

- 빌트인 타입을 써도 정의 파일은 만든다 (재사용·협업 품질·역할/스킬 분리 때문).
- **QA 에이전트는 general-purpose** — Explore는 읽기전용이라 검증 스크립트 실행 불가.
- model은 기본 opus. 추론이 약하면 하네스 전체 품질이 무너진다.

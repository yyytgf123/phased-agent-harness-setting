# Design Principles — 전 Phase 공통 설계 원칙

특정 단계가 아니라 키트 전체에 걸쳐 지켜야 하는 횡단 원칙.
각 phase README가 세부를 정하지만, 충돌 시 이 원칙이 우선한다.

> 이미 키트 곳곳에 반영된 원칙(권한 최소화·Human-in-the-loop, 규칙 최소화·CLAUDE.md 60줄,
> 도구 남발 주의, 완벽한 사전설계 지양·반복 개선)은 여기서 재서술하지 않는다.
> 아래는 그 원칙들을 **운영 규칙으로 못 박는** 추가 항목이다.

## 1. 말로 지시하지 말고 시스템으로 강제 (가장 중요)
"~하지 마라"는 프롬프트 문장은 확률적으로 무시될 수 있다.
**가능하면 구조적으로 실패를 차단**한다. 강제 수단 우선순위:

1. **권한/접근 차단** — 위험 명령을 애초에 실행 못 하게 (allowlist, 범위 밖 쓰기 금지).
2. **Hook (결정적 게이트)** — 비결정적 AI에 결정적 제어를 끼운다.
   - `PreToolUse` — 도구 실행 전 검증 (예: `git commit`/`terraform apply` 전 위험 명령 차단).
   - `PostToolUse` — 실행 후 처리 (예: 파일 변경 시 자동 린트/포맷).
   - `Stop` — "다 했습니다" 선언 전 ㄴ요구사항 대비 검증 강제 (조기 완료 방지).
3. **CI/린터** — PR 단계에서 구조적으로 거름 (spotless, tflint, kubeval, actionlint).
4. **프롬프트 텍스트 규칙** — 위 세 개로 못 막을 때의 최후 수단.

> 예: "console.log 지우라"고 CLAUDE.md에 쓰는 것보다, PreToolUse 훅에서
> 커밋 직전 `grep console.log` 발견 시 `exit 2`로 막는 게 안전하다(exit 1은 비차단=통과).
> 훅 설정 위치: `.claude/settings.json` + `.claude/hooks/*.sh`. 도구 인자는 stdin JSON으로 들어온다.

## 2. MCP/도구는 2~3개로 시작 (Tool Thrash 방지)
외부 API·DB·MCP를 무작정 연결하면 에이전트가 도구 선택에 인지 예산을 낭비하는
**Tool Thrash**가 생긴다. **2~3개로 시작하고, 실제 한계에 부딪힐 때만 추가**한다.
(이미 Phase 2의 tool cliff와 연결: 활성 도구 15개 이하 + 시작은 더 적게.)

## 3. 컨텍스트 상한 — 많을수록 좋은 게 아니다
무조건 많은 문맥을 넣지 않는다. **체감상 ~300K 토큰을 넘기면 결과 품질이 오히려 떨어진다.**
대응:
- 항상 로드되는 규칙은 최소화 (헌법 `claude.md`는 정책만 담고 ≤120줄, 깊은 내용은 docs/(SDD)로 — 도메인 규칙은 paths로 스코프 좁힘).
- 세부 지식은 스킬 references/로 내려 **필요할 때만** 로드 (Progressive Disclosure).
- 큰 산출물·로그는 컨텍스트에 쌓지 말고 `_workspace/` 파일로 외부화.
- 긴 작업은 subtask로 쪼개 컨텍스트가 누적되지 않게 한다.

## 4. Bug Log → 지속 학습 루프 (같은 실수를 시스템이 기억하게)
세션이 끝나면 AI는 다 잊는다. 반복 실수를 막는 가장 싼 장치가 Bug Log이고,
이를 "관찰 → 점수 → 승격 → 정리" 루프로 키운 게 자체 학습 시스템이다.
- 최소형: 헌법 `claude.md` 하단 `## Bug Log`에 실수마다 한 줄 `[날짜] 무슨 실수 → 어떻게 방지`.
- 고도화: Hook으로 자동 관찰 + 신뢰도 점수 + 프로젝트 스코프 + 승격/정리.
  → `evolution.md`에서 다룬다 (외부 의존 없이 markdown+훅으로 자체 구현).
- **매주 5분 리뷰**(산출물의 `/review`)로 패턴을 규칙으로 승격. (완벽 설계 대신 반복 개선.)

> Bug Log는 step5에서 `claude.md` 생성 시 빈 섹션으로 넣어두고,
> step7 검증·실사용에서 발견된 실수를 채워 나간다.
> 자동 관찰은 산출물 운영 단계에서 `.claude/instincts/`와 관찰 훅으로 켠다.

## 5. 에러 핸들링 (에이전트·오케스트레이터 공통 — 단일 소스)
실패 시 **1회 재시도 → 재실패면 누락을 명시하고 진행**한다. 에러 증거(로그·출력)는
삭제하지 말고 컨텍스트·`_workspace/`에 보존. 상충 데이터도 지우지 말고 출처를 병기한다.
도메인 특화 에러 표는 각 step이 이 원칙 위에 덧붙인다(예: `orchestrator-patterns.md`).

## 6. Progressive Disclosure (3단계 로딩 — 단일 소스)
항상 로드되는 걸 최소화하고 나머지는 미룬다: **메타데이터(항상) → 본문(필요 시) → 참조(요청 시).**
- claude.md(헌법)=항상 로드→정책만 담아 lean. skill description=항상→본문은 트리거 시→상세는 `references/`.
- skill 본문 ≤500줄, 초과분·도구별 변형은 `references/` 하위로 분리해 관련 파일만 로드.

## 7. 층위 주의 — 키트 지침 vs 실제 동작 (단일 소스)
`kb/*`·`steps/*` 문서는 **지침**이다. 규칙이 실사용에서 동작하려면 step5에서 루트
`claude.md`(상시 로드)·Hook·스킬에 **박혀야** 한다. 문서에만 있으면 새 세션엔 적용 안 된다.

## 8. 읽기 순서
각 phase는 **README.md** 부터 읽고 세부사항을 읽는다.

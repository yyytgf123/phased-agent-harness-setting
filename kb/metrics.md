# Metrics — 토큰·성능 측정 프로토콜

최적화의 효과를 **수치로 증명**하기 위한 측정 규칙. "줄였다"는 baseline 대비 delta로만 말한다.
도구는 `tools/token-report.sh`(외부 의존 없는 셸). 측정 철학은 README의 "도입 효과는 직접 측정"을 잇는다.

## 토큰을 어떻게 세나 (한계 명시)
- 토크나이저(tiktoken 등)가 없는 환경을 기본으로 본다. **1차 지표는 문자 수(chars)** — 같은 성격의
  텍스트끼리 전후를 비교할 때 가장 안정적인 프록시다(나눗수가 소거됨).
- 토큰 추정치는 `chars / TOK_DIV`(기본 4)의 거친 근사. KR/EN 혼합이라 **절대값은 부정확**하다.
  오직 **상대 비교(전/후)** 에만 쓴다. tiktoken이 있으면 그걸로 교체해도 결론은 같다(상대값 유지).

## 무엇을 재나 (3종 비용 분리)
| 구분 | 측정 대상 | 왜 중요 | 명령 |
|------|-----------|---------|------|
| 빌드타임 | 키트 docs(steps/kb/catalog/templates) | 하네스 1회 구축 시 로드 | `--build` |
| 런타임-상시 | 생성된 루트 `CLAUDE.md` | **매 작업마다** 항상 로드 → 최대 레버리지 | `--runtime <proj>` |
| 런타임-조건 | agent 파일 / skill body | 호출·트리거 시에만 로드(평균·최대) | `--runtime <proj>` |
| 중복률 | 반복 규칙의 등장 파일 수 | 단일소스+1줄참조로 수렴해야 | `--dup` |

## 임계(게이트 기준)
- 루트 `CLAUDE.md` ≤ **60줄** (항상로드 상한). 초과 시 스코프 references로 분리.
- skill `SKILL.md` body ≤ **500줄** (트리거 로드 상한). 초과분은 `references/`로.
- 반복 규칙(안전·에러·점수표 등)은 **권위 소스 1곳의 전문 + 나머지는 1줄 참조**.
  `--dup` 의 file 수가 늘어도 "전문 중복"이 아니라 "참조"면 허용.

## 언제 재나
1. **착수 전(baseline)**: `--build`·`--dup` 스냅샷을 `docs/metrics/baseline-YYYYMMDD.md`에 기록.
2. **각 Stage 적용 후**: 같은 명령으로 재측정, delta를 같은 파일에 누적.
3. **실구축 검증**: 대표 스택으로 하네스를 1회 구축한 뒤 `--runtime <proj>`로 런타임 비용 확인.
4. **상시(Phase 6/7)**: 검증 게이트·주간 리뷰에서 `--runtime`을 돌려 회귀를 감시.

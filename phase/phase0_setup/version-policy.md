# Version Policy — 버전 결정 규칙

사용자가 버전을 "미지정"으로 두면 이 규칙으로 호환성 좋은 버전을 정한다.
도구 버전·호환성은 시간이 지나면 바뀌므로 내장 지식만 믿지 말 것.

## 기본 동작 (version_policy: auto)
1. 내장 지식으로 호환 조합을 **제안**한다.
2. 다음이면 **웹 검색으로 확인** 후 확정:
   - 프레임워크 메이저 ↔ 언어 버전 호환 (예: Spring Boot 3.x ↔ Java 17+)
   - 최근 1~2년 빠르게 바뀌는 도구 (IaC provider, k8s, 런타임 LTS)
   - 사용자가 "최신" 또는 "LTS"를 명시
3. 확정 버전을 **표로 출력**, 각 버전에 근거(검색/LTS/호환매트릭스) 한 줄.
4. 사용자 확인 후에만 Phase 1로.

## strict 정책
명시된 버전만 사용. 미지정 항목은 임의로 정하지 말고 사용자에게 묻는다.

## 호환성 우선순위 (auto 판단 기준)
1. **언어 LTS 우선** (예: Java 21 LTS).
2. **프레임워크가 언어를 제약** — 프레임워크 요구 최소 언어 버전 확인 후 그 위에서 LTS.
3. **빌드툴은 프레임워크 권장 범위**.
4. **인프라는 provider 호환 매트릭스** — IaC는 provider/모듈 지원 범위의 안정 버전.
5. **k8s는 최신 -1~-2 마이너** — 최신 바로 아래 안정 버전.

## 출력 형식 (Phase 0.5 산출물)
```
| 도구 | 선정 버전 | 근거 |
|------|-----------|------|
| Java | 21 (LTS) | 검색 확인, Spring Boot 3.x 호환 |
| Spring Boot | 3.x 최신 patch | 검색 확인 |
| Gradle | 8.x | Spring Boot 3.x 권장 |
| Terraform | 1.x 안정 | provider 호환 매트릭스 |
| ... | ... | ... |
이 조합으로 진행할까요?
```

## 주의
- "최신" 결과를 그대로 쓰지 말고, 검색 시 현재 연도를 넣어 stale 결과 제거.
- 알려진 호환성 깨짐(breaking)이 있으면 표 아래 경고.

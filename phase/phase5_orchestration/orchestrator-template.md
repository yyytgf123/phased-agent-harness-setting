# Orchestrator Template — 개발/인프라

오케스트레이터는 "누가 언제 어떤 순서로 협업하는가"를 정의하는 특수 스킬.
개별 스킬이 "각 에이전트가 무엇을 어떻게"라면, 오케스트레이터는 "팀 전체 흐름".

## 모드별 패턴

**에이전트 팀 모드 (기본)**
```
[오케스트레이터/리더]
  ├── TeamCreate(team, members=[backend-dev, infra-dev, reviewer, qa?])
  ├── TaskCreate(tasks with dependencies)   # 의존: 구현→테스트→배포plan
  ├── 팀원 자체 조율 (SendMessage)
  ├── 결과 수집 및 종합
  └── 팀 정리
```

**서브 에이전트 모드**
```
[오케스트레이터]
  ├── Agent(security-check, run_in_background=true)
  ├── Agent(perf-check, run_in_background=true)
  ├── 결과 대기·수집
  └── 통합 산출물
```

## 데이터 전달 프로토콜

| 전략 | 방식 | 모드 | 적합 |
|------|------|------|------|
| 메시지 기반 | SendMessage 직접 통신 | 팀 | 실시간 조율, 피드백, 가벼운 상태 |
| 태스크 기반 | TaskCreate/Update | 팀 | 진행추적, 의존관리, 작업요청 |
| 파일 기반 | 약속된 경로 R/W | 둘 다 | 대용량, 구조화 산출물, 감사추적 |

**팀 모드 권장 조합:** 태스크(조율) + 파일(산출물) + 메시지(소통)

**파일 기반 규칙**
- 중간 산출물은 `_workspace/`에 저장
- 파일명: `{phase}_{agent}_{artifact}.{ext}` (예: `02_backend-dev_diff.md`, `03_infra-dev_plan.txt`)
- 최종 산출물만 사용자 경로로, 중간 파일은 보존(사후 검증·감사)

## 에러 핸들링 (개발/인프라 특화)

| 에러 유형 | 전략 |
|-----------|------|
| 테스트 실패 | 1회 재시도 → 재실패 시 실패 로그 보존하고 reviewer에 보고 (증거 삭제 금지) |
| `terraform plan` 차이 예상 밖 | 중단. 사람에게 plan 출력 제시. 절대 apply 금지 |
| 범위 침범 시도 | 거부 + 담당 에이전트로 위임 |
| 빌드 환경 문제 | tooling-matrix로 명령 재확인 후 1회 재시도 |
| 상충 데이터 | 삭제 말고 출처 병기 ("backend-dev: X / qa: Y") |

핵심 원칙: **1회 재시도 후 재실패 시 누락 명시하고 진행.** 에러 증거는 컨텍스트에 남긴다.

## 팀 크기 가이드라인

| 작업 규모 | 팀원 수 | 팀원당 작업 |
|-----------|---------|-------------|
| 소 (5~10 작업) | 2~3 | 3~5 |
| 중 (10~20) | 3~5 | 4~6 |
| 대 (20+) | 5~7 | 4~5 |

> 3명의 집중된 팀이 5명의 산만한 팀보다 낫다. 팀원↑ = 조율 오버헤드↑.

## 테스트 시나리오 (오케스트레이터에 포함)
- 정상 흐름 1개 (예: 엔드포인트 추가 → 테스트 통과 → plan 생성)
- 에러 흐름 1개 이상 (예: 테스트 실패 시 폴백, apply 차단 동작)
